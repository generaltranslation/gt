gt migrate: hold back conversions that cannot produce a working app, instead of writing them. Callers using next-intl's locale-aware navigation signatures (router.replace(href, { locale }), redirect({ href, locale })) skip and keep the navigation wrapper on next-intl; react-intl IntlShape bindings and react-i18next translation functions that escape as values (`.locale` reads, `translateKnown(t, ...)`) skip their files; async Server Components get `await getTranslations()` instead of a client hook; an unrecognized Next config either gets withGTConfig wrapped around its whole export (function configs included) or stops the run before anything is written; react-intl/react-i18next runs pre-flight the server provider boundary (a Server Component <body> layout, [locale] as the root layout) and stop with the manual steps when gt-next could not resolve the locale per-route; latent server-calls-client-hook violations withhold static-rendering restoration (build parity with the baseline) and are named in the report. Also: test files become an explicit manual stage in the report, dictionary-only runs emit `cacheUrl: null` (silences the missing-projectId warning), TypeScript wrappers re-export Link under next/link's concrete type (gt-next/link ships untyped props), and reports no longer list byte-identical writes as converted or claim GTProvider was missing when the root layout received it.

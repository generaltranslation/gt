`gt migrate` now supports `--from react-intl` and `--from react-i18next` in addition to `--from next-intl`, with sources handled by pluggable adapters (the value is validated against your installed dependencies).

react-intl (FormatJS): supported through the same dictionary-compat path — `useIntl().formatMessage({ id }, values)` (including the destructured `const { formatMessage } = useIntl()` form) and `<FormattedMessage id values />` become `t(id, values)`, the value-formatter components (`<FormattedNumber>`, `<FormattedDate>`, `<FormattedTime>`, `<FormattedPlural>`, `<FormattedRelativeTime>`) map to gt-next components, RSC `createIntl(...).formatMessage` becomes `await getTranslations()`, and FormatJS catalogs are reused (no format conversion, since gt shares FormatJS's ICU parser); flat dotted keys (`{"Home.title": …}`) are re-nested into new catalog files (`{Home:{title:…}}`) so gt-next's nested-path resolver can find them, and originals are never mutated. When only non-default-locale catalogs ship, a default-locale catalog is synthesized from harvested literal `defaultMessage`s. Unsupported react-intl APIs (class-component `injectIntl`, `<FormattedList>`, render-prop children, `<IntlProvider>` props beyond locale/defaultLocale/messages such as `timeZone`/`onError`, and others) are skipped and reported. Projects using FormatJS auto-generated ids (`defaultMessage` with no literal `id`; ids hashed at build time via `overrideIdFn`/`idInterpolationPattern`) are not converted in v1 — add explicit ids or convert those files by hand.

react-i18next: converts the i18next `public/locales/{lng}/{ns}.json` catalogs to ICU (per-locale CLDR plurals, ordinals, call-site-gated context, formatters, `$t()` nesting) merged into new `gt/dictionaries/{locale}.json` without touching the originals, swaps the provider and config, and migrates the call sites that import `useTranslation`/`Trans` directly from react-i18next (`useTranslation` → `useTranslations`, `t('ns:key')` remapped to dotted dictionary paths, `i18n.changeLanguage()` → `useSetLocale`, `<I18nextProvider>`/layout → `<GTProvider>`, trivial `<Trans>` → `t()`). Wrapper-based call sites (the official App Router pattern, where components import from a local `i18n/client`/`i18n/server` module), the bespoke server side (`getT`), and non-trivial `<Trans>` are reported and left for manual migration rather than miscompiled; next-i18next / bare i18next / Pages Router route out with a scoped message. This complements PR #1602 with no runtime dependency, and reserves a future `--keep-i18next-format` mode for i18next-native rendering.

Every skip and TODO is listed in the generated `gt-migrate-report.md`, and teardown stays scope-safe per source library.

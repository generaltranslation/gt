# Debug missing or broken translations

Use this when content renders in the source language only, a locale is empty, or the build
crashes around translation loading. Work through the checks in order; the first match is
usually the cause.

## 1. Is the content actually marked for translation?

- JSX must be inside a `<T>` component. Text outside `<T>` is never translated.
- Plain strings (a `placeholder`, `aria-label`, `alt`, or a value in code) are not
  translated by `<T>`. Use `useGT()` (client or sync server components) or `getGT()` from
  `gt-next/server` (async server components). For gt-react string helpers see its
  `AGENTS.md`.

## 2. Is the provider in place? (gt-react and gt-next client trees)

- gt-next: `GTProvider` must wrap the app at the root layout.
- gt-react: the app must be initialized (`initializeGT` for server-rendered React,
  `initializeGTSPA` for single-page apps) and, where required, wrapped in `GTProvider`.

## 3. Have translation files been generated?

If the project uses local translation files (`files.gt.output` set to a `_gt` folder):

- Run `npx gt translate`. Files under `public/_gt/` (gt-next) or `src/_gt/` (Vite) only
  exist after this runs, and the folder is gitignored, so a fresh clone has none.
- Make sure the build runs it: `"build": "npx gt translate && next build"`.

## 4. Does the loader crash or return nothing?

The `loadTranslations` function must default-export and its import path must match the
output directory. A common crash is a bare dynamic import of a file that does not exist yet:

```ts
// Falls back to source language before files are generated
export default async function loadTranslations(locale: string) {
  try {
    const t = await import(`../public/_gt/${locale}.json`); // match your output path
    return t.default;
  } catch {
    return {};
  }
}
```

- The loader must be the default export or a named `loadTranslations` export. If neither
  is present, development throws an unresolved-loader error.
- Wrong path (for example `../translations/` when output is `public/_gt`) returns nothing
  or throws.

## 5. Dictionary lookups

If using `useTranslations`/`getTranslations`, the error `Dictionary entry <id> cannot be
found` means the id is not in the dictionary. Confirm the id exists and that the dictionary
is registered (via the `dictionary` option in `withGTConfig` for gt-next, or the
`dictionaries` prop on `GTProvider` for gt-react).

## 6. Environment and keys

- Development on-demand translation needs a dev key. gt-next reads `GT_DEV_API_KEY` (or
  `NEXT_PUBLIC_GT_DEV_API_KEY`) and `GT_PROJECT_ID`. A Vite gt-react app reads
  `VITE_GT_DEV_API_KEY` and `VITE_GT_PROJECT_ID`.
- A production key (`gtx-api-`) used as a dev key, or a dev key present with
  `NODE_ENV=production`, causes errors. Match the key type to the environment.

## 7. The language will not switch

The selection is stored in the `generaltranslation.locale` cookie. A stale cookie
overrides the dropdown. Clear cookies and retry.

## 8. It works but is slow in development

Expected. Development translates on demand through the API. Production is pre-generated by
`gt translate` and loads instantly.

## Confirm the fix

Run `npx gt validate` and load the app in the target locale.

# Validate before committing an i18n change

Use this before committing changes that touch translated content, `<T>` usage, dictionary
entries, or GT config. It catches broken markup and out-of-sync ids early.

## Steps

1. Run the validator. It scans the project (dictionary and inline content) and reports
   errors:

   ```bash
   npx gt validate
   ```

   To check specific files only, pass them: `npx gt validate src/app/page.tsx`.

2. Fix anything it reports. Common issues it surfaces:
   - Malformed `<T>` content or unsupported nesting.
   - Dictionary ids referenced in code but missing from the dictionary.
   - Configuration problems in `gt.config.json` or the plugin options.

3. Do not commit generated translation output as if it were source. The `_gt` output
   folder (`public/_gt/` for gt-next, `src/_gt/` for Vite) is normally gitignored and is
   regenerated by `npx gt translate`. Confirm it is in `.gitignore` and that translation
   runs in the build (`"build": "npx gt translate && next build"`).

4. Confirm no API key is being committed or exposed. `GT_API_KEY` is a production secret,
   belongs in the CI/CD environment only, and must never be prefixed with `NEXT_PUBLIC_`
   or `VITE_`.

## Verify

A clean `npx gt validate` run and a review of the diff (no committed secrets, no committed
`_gt` output) means the change is ready to commit.
